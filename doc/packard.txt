1. Introduction                              *packard-introduction*

packard.nvim is a security-first Neovim plugin manager built on the native
|vim.pack| module (Neovim 0.12+).  It protects against supply chain attacks
by enforcing a configurable cooldown period on new commits and requiring
manual user review before any plugin is updated.

packard uses a commit-pinning model (via `nvim-pack-lock.json`) and features
parallel git fetch, inline AI diff review, semver version constraints, and a
lazy.nvim-style dashboard.

Requirements ~
  - Neovim >= 0.12.0
  - `git` and `curl` in your PATH

2. Setup                                        *packard-setup*

>lua
  require("packard").setup({
    defaults = { minimum_release_age = 30 },
    plugins = {
      "neovim/nvim-lspconfig",
      { "tpope/vim-fugitive", cmd = "Git" },
    },
    -- specs_dir = "lua/plugins",   -- alternative: directory-based specs
    -- debug = true,                 -- verbose logging via :messages
    -- notifications = true,         -- notify on eligible reviews
    -- self_management = true,       -- packard manages itself (default)
    -- highlights = {},              -- custom highlight overrides
    -- ai_review = {},               -- optional AI review config
  })
<

Options ~

  defaults ~
    Table of global defaults applied to all plugins:
      minimum_release_age    Number, cooldown in days (default 30).  Set to 0
                             to disable cooldown for all plugins.

  plugins ~
    Array of plugin specs (see |packard-spec|).  Each entry is a string
    `"owner/repo"` or a table `{ "owner/repo", ... }`.

  specs_dir ~
    String, path to a directory of `.lua` spec files.  Relative paths
    resolve against `stdpath('config')`.  Files are loaded via |loadfile()|,
    merged with the inline `plugins` array.  See |packard-spec-file|.

  debug ~
    Boolean, enables verbose tracing to `:messages`.

  notifications ~
    Boolean, when true (default) shows a startup notification if any plugins
    are eligible for review.

  self_management ~
    Boolean, when true (default) prepends `"ruicsh/packard.nvim"` to the
    plugin list so packard manages its own updates.  Set to false to manage
    packard manually.

  highlights ~
    Table of highlight group overrides (see |packard-highlights|).

  ai_review ~
    Table with AI review configuration (see |packard-ai|).

3. Plugin Spec                                  *packard-spec*

Each plugin entry is a string `"owner/repo"` or a table:

>lua
  { "owner/repo",
    -- spec fields (all optional)
  }
<

Supported fields ~

  name                String, override the plugin name derived from the repo.

  dir                 String, use a local filesystem directory for the plugin.
                      Mutually exclusive with an `"owner/repo"` source.
                      The plugin must already exist on disk.

  branch              String, git branch to track.  Default is the remote
                      HEAD (discovered via `git ls-remote --symref`).

  commit              String, exact commit SHA (40-char hex).  Highest
                      priority pin.

  tag                 String, exact git tag to pin.

  version             String, semver range (e.g. `"1.*"`, `"^2.0"`,
                      `"~1.2.3"`, `">=1.0"`).  Resolves to the latest
                      matching git tag.  Pre-release tags (-beta, -rc) are
                      excluded by default.

  minimum_release_age Number, per-plugin cooldown override in days.
                      Overrides `defaults.minimum_release_age`.

  priority            Number, load order (higher = earlier).  Processed
                      within the topological sort of dependencies.

  enabled             Boolean or function returning boolean.  When false,
                      the plugin is excluded entirely.

  cond                Boolean or function returning boolean|string.
                      When false the plugin is installed but NOT loaded.
                      Use for conditional loading.

  init                Function called at setup() before the plugin loads.
                      Receives `(plugin)`.  Useful for `vim.g.*` setup.

  config              Function or boolean.  Function: called after the
                      plugin loads as `function(plugin, opts)`.  Boolean
                      true: auto-calls `require(MAIN).setup(opts)`.

  opts                Table or function returning table.  Options passed
                      to the `config` function.  If `config` is absent
                      but `opts` is present, `require(MAIN).setup(opts)`
                      is called automatically.

  main                String, override the auto-detected Lua module name
                      used by `config`/`opts` auto-setup.

  keys                String, string[], table, or function.  Keymaps
                      registered immediately after the plugin loads.
                      First-wins conflict handling: existing keymaps are
                      never overwritten.

  cmd                 String or string[].  User commands registered
                      immediately after the plugin loads.

  dependencies        String, string[], or table[].  Plugins required by
                      this plugin.  Strings are parsed as specs; tables
                      support the full spec format.

  build               Function, string (":Command", "*.lua", or shell),
                      string[], or false.  Post-install/update build step.
                      Auto-detects `build.lua` / `build/init.lua`.

  ai_review           Table, per-plugin AI config override.  See |packard-ai|.

Pin priority:                        *packard-pin-priority*
  commit > tag > version > branch > default branch HEAD

Directory-based spec files:          *packard-spec-file*

  Instead of declaring all plugins inline, organize them across `.lua` files:

>lua
  require("packard").setup({
    specs_dir = "lua/plugins",
  })
<
  Files are loaded via |loadfile()| (not |require()|) to avoid caching.
  Parent directories are scanned before subdirectories.  Files starting
  with `_` are skipped.  Each file returns a single spec or an array of
  specs.

  File specs and inline `plugins` are merged.  For duplicate `owner/repo`:
  - keys, cmd, opts, dependencies are deep-merged
  - Other fields: last occurrence (inline) wins

Duplicate spec merging:               *packard-spec-merge*

  When multiple specs for the same plugin are declared:
  - keys: combined into a single list (wrapped in function if needed)
  - opts: deep-merged (supports sub-module composition)
  - cmd, dependencies: combined into a list
  - Other fields: last occurrence wins

4. Commands                                  *packard-commands*

                                                   *:Packard*
  :Packard           Open the dashboard to the Installed tab.

                                                   *:Packard-check*
  :Packard check     Run parallel update check: git fetch all plugins,
                     detect new commits, register in cooldown queue.
                     Echo summary with eligible/cooldown counts.

                                                   *:Packard-review*
  :Packard review    Open the dashboard to the Pending tab.

                                                   *:Packard-summary*
  :Packard summary   Open the dashboard to the Summary tab (update history).

                                                   *:Packard-clean*
  :Packard clean     Open the dashboard to the Clean tab for orphan
                     management (uninstalled plugin directories and stale
                     state metadata).

                                                   *:Packard-build*
  :Packard build [name]
                     Rebuild a plugin with a build step.  If no name is
                     given, rebuild all plugins with build steps.

                                                   *:Packard-help*
  :Packard help      Open the dashboard to the Help tab showing all
                     keybindings.

                                                   *:checkhealth-packard*
  :checkhealth packard
                     Report packard health: config, lockfile, pending
                     queue, AI config, network, plugin directories.

5. Dashboard Keybindings                      *packard-dashboard*

The dashboard shows 6 tabbed views.  Keybindings apply buffer-locally:

  Key               Action
  I                 Switch to Installed tab
  U                 Run update check and switch to Update tab
  P                 Switch to Pending tab
  S                 Switch to Summary tab
  C                 Switch to Clean tab
  ?                 Switch to Help tab

  j / k             Move cursor up / down
  <CR>              Show commit log inline (Installed/Pending tabs)
  <Space> / x       Toggle selection (Clean tab)

  a                 Approve pending update (Pending tab, eligible only)
  A                 Force approve (override cooldown)
  X                 Reject pending / Clean selected orphans

  gx                Open forge compare URL in browser (Pending tab)
  r                 Toggle AI review inline (Pending tab)
  R                 Force re-run AI review (bypass cache)

  B                 Rebuild plugin under cursor

  q / <Esc>         Close dashboard

6. Highlight Groups                           *packard-highlights*

All highlight groups are defined with `default = true` so your colorscheme
can override them.  Override via the `highlights` option in |packard-setup|:

>lua
  require("packard").setup({
    highlights = {
      PackardHeader = { link = "Title" },
      PackardEligible = { link = "DiagnosticOk" },
    },
  })
<

Groups ~

  Group                           Default Link               Usage
  PackardButton                   CursorLine                 Tab pill (inactive)
  PackardButtonActive             Visual                     Tab pill (active)
  PackardKeyHint                  @punctuation.special       Tab shortcut key

  PackardH2                       Comment                    Section header
  PackardComment                  Comment                    Subtle text

  PackardPluginName               Normal                     Plugin owner/repo
  PackardPluginNameSelected       Visual                     Plugin name (selected)
  PackardRowSelected              Visual                     Selected row background

  PackardCommitHash               Identifier                 Commit SHA
  PackardCommitAge                (none)                     Commit age text

  PackardEligible                 Normal                     Eligible badge / status
  PackardCooldown                 Normal                     Cooldown status

  PackardAIRiskLow                DiagnosticInfo             AI risk: low
  PackardAIRiskMedium             DiagnosticWarn             AI risk: medium
  PackardAIRiskHigh               DiagnosticError            AI risk: high
  PackardAIValue                  (none)                     AI field value text

  PackardStatusWarn               DiagnosticWarn             Warning / fallback status

  PackardProgressDone             Normal                     Progress bar done
  PackardProgressTodo             Normal                     Progress bar remaining

  PackardHelpSection              Title                      Help section title
  PackardHelpKey                  Special                    Help keybinding key
  PackardHelpDesc                 Comment                    Help keybinding description
  PackardHelpIntro                Comment                    Help intro text

7. AI Review                                    *packard-ai*

packard can optionally review plugin diffs using an LLM provider for
security risk analysis and breaking change detection.

Configuration ~

Add an `ai_review` table to |packard-setup|:

>lua
  ai_review = {
    provider = "openai",               -- "openai", "anthropic", "ollama", "custom"
    model = "gpt-4o",                  -- model identifier
    url = "https://api.openai.com/v1/chat/completions",  -- endpoint (optional, has defaults)
    headers = {
      ["Authorization"] = "Bearer " .. (os.getenv("OPENAI_API_KEY") or ""),
    },
    diff_warn_kb = 50,                 -- warn before sending diffs > 50 KB
    diff_error_kb = 200,               -- block diffs > 200 KB
  }
<

Provider default URLs ~

  Provider       Default URL
  openai         https://api.openai.com/v1/chat/completions
  anthropic      https://api.anthropic.com/v1/messages
  ollama         http://localhost:11434/api/chat
  custom         (required — no default)

Set `url` to override the default (useful for OpenAI-compatible proxies,
local endpoints, or custom gateways).

Trigger in the dashboard ~

  r   Toggle AI review on the current pending row
  R   Force re-run AI review (bypass the local cache)

Results are cached in `stdpath('state')/packard-ai-cache.json` by
`owner_repo:from_sha:to_sha` key.

8. Colorscheme Autoload                      *packard-colorscheme*

ADR-011: packard registers a `ColorSchemePre` autocmd during setup to
auto-load colorscheme plugins on demand.

When an unloaded (|cond| = false) plugin ships a colorscheme file at
`colors/<name>.{lua,vim}`, running `:colorscheme <name>` triggers packard
to load the plugin — including its `init()` hook, `config()`, keys, and
commands — before Neovim applies the scheme.

No spec field is required.  Any plugin with a `colors/` directory is
eligible.  Built-in Neovim colorschemes are short-circuited via
|getcompletion()|.

Use `cond = false` on colorscheme plugins you want installed but not
loaded until you switch to them:

>lua
  {
    "catppuccin/nvim",
    cond = false,
    opts = { flavour = "mocha" },
  }
<
`:colorscheme catppuccin` will auto-load the plugin, run its setup, and
apply the scheme.

9. Security Model                            *packard-security*

packard protects against supply chain attacks through a multi-gate workflow:

  1. Discovery
     `:Packard check` fetches the latest HEAD for each plugin via parallel
     `git fetch`.  New commits are detected by comparing against the
     installed SHA recorded in `nvim-pack-lock.json`.

  2. Quarantine
     Each new commit enters a "Pending" queue with a discovery timestamp.
     No code runs at this stage.

  3. Cooldown (default 30 days)
     Commits are held in "In Cooldown" until `minimum_release_age` days
     have passed since discovery.  During this window, other users may
     discover if the commit is compromised.

  4. Audit
     The user reviews eligible changes via:
     - `gx` to open a forge compare URL (GitHub, GitLab, Bitbucket) in the browser
     - `r` to run an inline AI security analysis of the diff
     - `<CR>` to view the local commit log

  5. Action
     a   Approve: the commit is applied via `vim.pack.update()`
     X   Reject (blacklist): the commit is permanently blocked
     A   Force approve: bypass remaining cooldown

Key security properties ~

  - No plugin code is updated without explicit user approval.
  - SSH URLs are rejected at spec parse time — public repos only via HTTPS.
  - AI diffs are sent to the configured provider; packard stores no API keys.
  - packard manages itself through the same cooldown + review flow.
  - The lockfile (`nvim-pack-lock.json`) lives in |stdpath-config| and is
    naturally version-controlled with your dotfiles.
  - Machine-local state (`packard-state.json`, `packard-ai-cache.json`)
    lives in |stdpath-state| — not VCS tracked.
